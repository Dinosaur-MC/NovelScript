# 项目开发元指令

## 议题介绍

### AI 小说转剧本工具

- 很多小说作者希望将自己的作品改编成剧本，请开发一款 AI 辅助剧本创作工具，降低改编门槛，提升效率。
- 要求：能将 3 个章节以上的小说文本自动转换为结构化剧本（YAML 格式），让作者可以快速获得可编辑、可进一步打磨的剧本初稿。请额外写一篇文档，定义剧本的 YAML Schema。文档中需说明该 Schema 的设计原因。

## 评审&提交规则

围绕作品质量、开发过程与演示表达综合评审，并按规范提交作品材料。

### 评审规则

- 作品完整度与创新性（40%）：产品设计合理性、功能完整度、交互流畅度、是否新颖有创意。
- 开发过程与质量（40%）：架构清晰度与合理性、代码健壮度-逻辑、规范、可读性等，PR数量与质量、commit分布合理性。
- 演示与表达（20%）：demo视频清晰完整度、是否清晰完整表达作品功能和效果。

### 作品提交规范

#### 一、须提交内容

1. 公开的github/gitee仓库，提交时间截止后可设置为可公开访问。
2. demo视频。
3. README文档。

#### 二、作品有效性

1. 作品需符合所选议题方向，并确保自主完成无抄袭。
2. 全周期持续交付，严禁临尾“突击提交”。从议题发布之日起，开发周期内保持持续的PR（Pull Request）记录和 commit 提交。仅在最后一天一次性导入所有代码的作品，将直接视为无效作品。
3. 所有commit 时间戳必须落在所选批次的开始与截止时间之内，否则视为无效作品。
4. 其他无效情形：
	- PR 描述空白或与实际代码变更严重不符。
	- 引用了第三方库或框架，没在 README 中列明依赖，且未说明原创功能部分。
	- 复用了自己过去的代码片段，没在 PR 描述中注明来源。

#### 三、PR（Pull Request）提交规范

1. 请基于 PR 添加新功能。
2. 每个PR只做一件事：每个 PR只实现或修改单一功能；鼓励尽可能小、粒度尽可能细的 PR；大功能应拆分为多个独立 PR 分步提交。
3. PR标题与描述需清晰完整，内容包含：
	- 标题：一句话说明本 PR 新增/修改了什么。
	- 功能描述：说明该功能的作用与使用方式。
	- 实现思路：简要说明技术选型或核心实现逻辑。
	- 测试方式：如何验证该功能正常运行。
4. PR合并后，主分支代码需保持可运行状态，评委在任意时间查看应能复现演示效果。

> 可通过 `gh` 工具管理 GitHub 仓库和 PR。

#### 四、demo视频要求

1. 请用声音完整讲解、展示作品主要功能和效果，覆盖核心模块。
2. 请上传至可访问的外部平台例如 bilibili/云盘等，链接放到代码仓库的README中，视频可播放。

#### 五、其他要求

1. 多人组队，每队仅需提交一个代码仓库地址。队员均须使用各自账号提交commit，PR清晰描述各自分工。
2. 如项目包含多个独立模块，应提交至同一仓库的不同子目录（如 /frontend 与 /backend）分别管理，以便于部署和启动。
3. 代码仓库需在开题后创建。
